하는 것

html, javascript, css로 기초를 닦고 bootstrap으로 grid design, id와 class로 생각하던것 보다 훨씬 더 많은것을 할 수 있다는것을 깨닳음 sass와 emmet으로 한층 더 편리하게 사용

jquery로 dom을 이해, react를 공부하며 front가 굴러가는 구조에 대해 알아갔다. canvas, image, video, web audio context api, immutable, eslint등으로 자신에 대한 경쟁력 그리고 자신감을 상승시킴 요즘은 워낙 세상이 좋아져서 뭔가 필요하면 검색하여 찾고(왠만한건 다 있음), 없으면 만들면 됨

front-end 쪽만해도 공부해야 할 것이 산더미이기 때문에 back-end쪽은 많은 신경을 못썼고, 개인프로젝트를 하기위해선 back-end 자동화 framework meteor를 이용하여 진행. 회사에서 meteor -> laravel로 변환하며 back-end MVC에 대해 조금은 이해. 이후 업무를 위해서는 laravel 공부는 필수적이었고 프론트-백엔드 가릴 것 없이 업무에 필요한 기술을 누적하여 배워감.

하고 싶은 것

react와 더불어 좀 더 하드하게 사용 할 front-end library (angular, vue아직까진 토이용) 아직은 무엇을 만들때 react를 기반으로 component들이 머리속에서 떠오르지만 자유자재로 이용 할 수 있는 front-end library의 선택지가 많아지면 용도에 맞게 적용할 수 있을 것 같음음 그리고 redux를 코어하게 자유자재로 사용하고 싶은 마음 (따로 공부했기 때문에 토이같은 소규모에 적용해봄)

back-end와 database에 대해 조금 더 공부 해보고 싶음. 개발에 대한 자신감이 있다가도 없어지는게 서비스 전반적인 구조와 설계를 파악하지 못해서 일때가 많은것 같음. front-end에 필요한 data들이 어떻게 정제되서 오고, 그 데이터들은 결국 database에서부터 나오기 때문에 database 설계가 참 중요한 부분이라고 생각됨. 그래서 더 알아야한다고 생각.

개발자가 되고 난 뒤

개발자가 되고.. 더 정확히 말하면 이전 회사(tubloo)를 다니며 행복했고, 단 한번도 월요일이 두려운적이 없었다. 존경하는분들 밑에서 인정받으며 일하며 나 자신에 대한 생활 패턴도 많이 바뀌었음. 다른 업종에서 일 할때는 매일 아침 힘들고 늦잠에 지각을 밥먹듯이 했었고, 책은 근처에 두지도 않았으며 돈에 대한 개념도 없었지만, 지금은 시간을 나누어 새벽에 일어나 공부하는 그룹에 가입하여 출근 3시간 정도전에 자발적으로 집 & 커피숍에서 업무나 공부, 가계부정리등을 필요한 것을 한 뒤 출근.

개발문서나 동영상들이 왠만한건 영어로 되어있어서 3개월전 쯤부터 영어 공부에 미쳐 영어 그룹 스터디도 하며 공부중인데 역시나 영알못에게 쉽지 않은 도전. 그래도 예전보단 stackoverflow의 문맥을 더 이해 할수 있음. react공부 할 시기엔 react모임을 참석하여 약 6개월간 스터디. 어쩌면 첫 회사에선 개발보다 현상에 대해 사고하고 고민하고 상상하는 방법을 훨씬 더 많은것을 배움. 그것은 나의 인생에 아주 큰 부분이 되었음.

구구절절

개발을 시작했을때(그땐 몰랐지만..) web은 변화하고 있었고, javascript는 대격변이 진행되고 있었음. 조금 더 빠른, 조금 더 편한, 조금 더 효율적인 tool과 언어, 수 많은 library들이 쏟아져 나와 개발자는 서비스를 만들기 위한 수단을 입맛에 맛게 선택하는게 가능한 시기가 도래함.

처음 개발을 배웠던 2015년.. jquery못하면 취업못한다는 말이 있었는데, 어느새 jquery를 쓰면 취업못한다는 말까지 나오는 현실. 단순히 html, css퍼블리싱, button onclick event 연결하던 일이라고 배웠었지만 지금은 front-back-infra등 서비스의 전반적인 구조와 설계를 알아야 한다고 생각함. 물론 전문적인 back-end, infra 서버구축, 설계는 못하더라도 back-infra단에서 어떤 일이 발생했을때, 그 문제가 front-end에 미치는 영향이 무엇이고, 어떤 것을 대비 해야하는지 이해 할 수 있음. 신입 개발자 시절엔 근본없는 자신감이 있었고 1년차엔 넘어야 할 산을 하나하나 넘어 가는것이 너무 재밋었으며 지금은 근자감에 가득 차있다. 근본 있는 자신감. 자만심이나 나 자신의 실력을 과대평가 하는것이 아닌 항상 겸손의 자세를 가지고 앞으로 나아갈 길에 대한, 늘 그랫듯 할 수 있을 것이란 자신감이다.

TTD에 대한 생각

 말로는 그다지 어렵지 않다. 정확한 프로그래밍을 디자인 단계에서 반드시 미리 정의해야할 것들을 실제 코드 구현 이전 단계에 정의해서 오류와 예외 상황을 최소화하고 이렇게 단위 테스트가 쌓이게 되면 장기적으로 fdd(faith driven development)를 벗어날수있다. 하지만, "좋아! 이젠 정말 tdd뿐이야!" 하고 두번 정도 시도 해본적이 있다. 이틀정도는 잘 썻고 뿌듯했다. gulp에서 특정 keyword로 complie하면 자동으로 모든 유닛test를 거치도록도 만들어 보고 희망찬 미래를 그렸다. 하지만 몇일이 지나고 일이 쌓이고 긴급수정과 우선순위가 높은 작업들을 하다보니 마음속으론 어느새 tdd가 다시 fdd로 변했고 잘 되겠지? 별일 없을꺼야로 바뀌어 갔다. 그리고선 tdd가 맘에 안들기 시작했다.

 몇몇 글을 보면 tdd는 위대하며 tdd를 하지 않는 개발자는 자신의 코드에 대해 자신이 없다는등, 좋지 않은 개발자라는등 말이 들었다. 의욕에 넘쳐 tdd에 대한 글을 읽고 jest깔고 mocha등등 좋은 test도구를 찾아 설치 후 예제를 따라 열심히 하지만 다음날이 되면, 다음 일정이 오면 아tdd해야되는데 해야되는데 라고 말하지만 오늘은 아니라고 말한다. 나도 마찬가지다.. 아직까진 필요성을 느끼지 못하는것 같다.

 tdd가 좋다라는 의견엔 누구도 이견이 없을 것이다. 하면 반드시 좋다라는 것도 당연히 알고 있다. 하지만 난 아직 게으른 것 같다. 다들 좋다고 하는데엔 반드시 이유가 있고 나같이 급한 성격을 가진 사람에겐 언젠간 반드시 넘어야 할 산이다.
